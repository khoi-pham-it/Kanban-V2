\chapter{Phân tích và Thiết kế hệ thống}

\section{Mô tả bài toán và Tầm quan trọng}
Trong kỷ nguyên số, làm việc nhóm (Teamwork) và quản lý tiến độ dự án (Project Management) là những yếu tố quyết định sự thành bại của một tổ chức. Hệ thống quản lý công việc dạng danh sách truyền thống thường dẫn đến sự quá tải thông tin và khó hình dung tổng quan.

Mô hình Kanban - bắt nguồn từ hệ thống sản xuất tinh gọn của Toyota, cung cấp một giải pháp trực quan hóa quy trình làm việc hoàn hảo. Trạng thái của từng công việc (Task) được thể hiện bằng vị trí của nó trên một Bảng (Board). Ví dụ phổ biến nhất là luồng làm việc 3 bước: \textbf{Todo (Cần làm) $\rightarrow$ Doing (Đang làm) $\rightarrow$ Done (Hoàn thành)}.

Bài toán đặt ra cho hệ thống Trello Mini là phải số hóa được trải nghiệm Kanban vật lý này, sao cho người dùng có cảm giác như đang cầm một tờ giấy note (Card) và dán nó sang một cột (List) khác trên tấm bảng trắng (Board). Mọi thao tác phải diễn ra mượt mà, tức thời, không có độ trễ tải trang.

\section{Xác định yêu cầu tính năng (Use Cases)}
Hệ thống được thiết kế với các nhóm chức năng cốt lõi sau:
\begin{itemize}
    \item \textbf{Nhóm quản lý Phiên làm việc (Authentication):} Cung cấp tính năng Đăng nhập/Đăng xuất đơn giản. Thiết lập cơ chế \textit{Protected Routes} để đảm bảo tính riêng tư: Chỉ những người dùng đã xác thực (Authenticated) mới được phép truy cập vào khu vực Dashboard và Board Detail. Nếu cố tình truy cập, hệ thống tự động điều hướng (redirect) về trang đăng nhập.
    \item \textbf{Nhóm quản lý Bảng dự án (Board Dashboard):} Đóng vai trò là "Trang chủ" của không gian làm việc. Người dùng có thể xem danh sách tất cả các Dự án dưới dạng thẻ lưới (Grid view), tạo mới một dự án hoặc xóa bỏ một dự án đã hoàn thành.
    \item \textbf{Nhóm quản lý Cột trạng thái (List Management):} Trong một Bảng, người dùng có thể tạo ra vô số các Cột (Ví dụ: Backlog, In Progress, Reviewing, Done). Mỗi cột có thể được đổi tên hoặc xóa bỏ toàn bộ khi không còn nhu cầu.
    \item \textbf{Nhóm quản lý Thẻ công việc (Card Management):} Đây là hạt nhân của hệ thống. Mỗi thẻ đại diện cho một đầu việc cụ thể. Người dùng có thể thêm thẻ mới vào cuối danh sách, mở thẻ để xem/chỉnh sửa chi tiết (Mô tả, Nhãn dán - Labels, Ngày hết hạn), và xóa thẻ.
    \item \textbf{Nhóm tương tác Kéo Thả (Drag \& Drop):} Tính năng mang tính biểu tượng của Kanban. Người dùng có thể dùng chuột (hoặc ngón tay trên màn hình cảm ứng) nhấp giữ một Card và kéo nó thả sang một List khác để cập nhật trạng thái làm việc một cách trực quan nhất.
\end{itemize}

\section{Phân tích Feature-Sliced Design (FSD) trong thực tế}
Kiến trúc Feature-Sliced Design (FSD) được áp dụng triệt để trong dự án Trello Mini nhằm module hóa mã nguồn, giúp hệ thống dễ bảo trì và mở rộng khi quy mô tăng lên. Cấu trúc thư mục được chia thành các lớp (layers) rõ ràng theo mức độ phụ thuộc giảm dần:
\begin{itemize}
    \item \textbf{app}: Chứa cấu hình khởi tạo ứng dụng, routing tổng, global styles và các providers.
    \item \textbf{pages}: Chứa các trang chính (Login, Dashboard, BoardDetail), đóng vai trò ghép nối các features và entities lại với nhau để tạo thành góc nhìn hoàn chỉnh.
    \item \textbf{widgets}: Các khối giao diện phức tạp, độc lập và có khả năng tái sử dụng cao (ví dụ: Sidebar, Header).
    \item \textbf{features}: Chứa logic nghiệp vụ cụ thể mang lại giá trị cho người dùng (như tính năng kéo thả \texttt{DragAndDrop}, thêm thẻ mới \texttt{AddCardForm}).
    \item \textbf{entities}: Định nghĩa cấu trúc dữ liệu cốt lõi (Interfaces cho Board, List, Card) và chứa Zustand store quản lý state của các thực thể này.
    \item \textbf{shared}: Các thành phần dùng chung ở mức độ thấp nhất (UI components cơ bản như Button, Input; các hàm tiện ích utils; custom hooks).
\end{itemize}

\section{Mổ xẻ mã nguồn (Code Walkthrough)}

\subsection{Khởi tạo Zustand Store với Persist Middleware}
Việc lưu trữ dữ liệu cục bộ nhằm tránh mất mát khi tải lại trang được tự động hóa hoàn toàn thông qua \texttt{persist} middleware của Zustand. Mã nguồn được thiết kế gọn gàng, tách biệt phần khai báo logic khỏi UI:
\begin{lstlisting}
import { create } from 'zustand';
import { persist } from 'zustand/middleware';
import { produce } from 'immer';

export const useBoardStore = create(
  persist(
    (set) => ({
      boards: [],
      addCard: (listId, cardData) => set(produce((draft) => {
        // Logic tìm list và push card mới vào mảng với Immer
        const list = draft.boards[0].lists.find(l => l.id === listId);
        if (list) list.cards.push(cardData);
      })),
    }),
    { name: 'trello-storage' } // Tự động lưu xuống LocalStorage với key này
  )
);
\end{lstlisting}

\subsection{Luồng xử lý Kéo Thả (Drag \& Drop)}
Logic phức tạp nhất của dự án nằm ở việc tính toán vị trí sau khi người dùng thả thẻ. Khi sự kiện \texttt{onDragEnd} của \texttt{dnd-kit} được kích hoạt, hệ thống sẽ kiểm tra xem thẻ được thả trong cùng một cột (reorder) hay sang cột khác (move) để áp dụng thuật toán hoán đổi tương ứng:
\begin{lstlisting}
const handleDragEnd = (event) => {
  const { active, over } = event;
  if (!over) return; // Rơi ra ngoài vùng thả hợp lệ

  const activeId = active.id;
  const overId = over.id;

  // Gọi action từ Zustand store để tính toán lại mảng
  if (activeListId === overListId) {
    reorderCardInList(activeListId, activeId, overId);
  } else {
    moveCardBetweenLists(activeListId, overListId, activeId, overId);
  }
};
\end{lstlisting}

\subsection{Tối ưu hóa UI/UX bằng Tailwind CSS}
Để giao diện mượt mà và linh hoạt trên nhiều kích thước màn hình, Tailwind CSS được sử dụng triệt để. Các tiện ích như \texttt{flex-grow}, \texttt{overflow-y-auto} kết hợp với tùy chỉnh thanh cuộn (custom scrollbar) giúp tái tạo chính xác cảm giác cuộn mượt mà của Trello gốc:

\section{Luồng dữ liệu (Data Flow Diagram)}
Hệ thống quản lý trạng thái của Trello Mini tuân thủ nghiêm ngặt nguyên lý luồng dữ liệu một chiều (One-way Data Flow), đảm bảo tính dễ đoán (predictable) và dễ gỡ lỗi (debug):
\begin{enumerate}
    \item \textbf{Tương tác (Interaction):} Người dùng thực hiện thao tác trên giao diện (ví dụ: nhấp nút "Xóa thẻ").
    \item \textbf{Gọi Hành động (Dispatch Action):} Event handler trên Component sẽ gọi một hàm Action được cung cấp bởi Zustand Store (VD: \texttt{deleteCard(cardId)}).
    \item \textbf{Cập nhật Trạng thái (State Mutation):} Hàm Action sử dụng \texttt{Immer} để thay đổi một bản nháp (draft) của State, sau đó Zustand tự động hợp nhất (merge) bản nháp này thành State chính thức mới (đảm bảo tính bất biến - Immutability).
    \item \textbf{Phản hồi Giao diện (Re-render):} Các Component đang "đăng ký" (subscribe) theo dõi phần State bị thay đổi sẽ tự động được React tính toán lại và cập nhật DOM.
    \item \textbf{Lưu trữ ngầm (Persistence):} Ngay sau khi State thay đổi, Persist Middleware sẽ chạy ngầm để đồng bộ hóa chuỗi JSON đại diện cho cây State mới xuống LocalStorage của trình duyệt.
\end{enumerate}

\section{Kiến trúc Component (Component Tree)}
Theo nguyên lý Feature-Sliced Design, cây Component của dự án được cấu trúc lồng nhau một cách có hệ thống, đi từ lớp tổng quát ngoài cùng (Router/Layout) vào sâu tới các thành phần hiển thị nhỏ nhất (Thẻ, Form):
\begin{lstlisting}
App
 +- RouterProvider
    +- MainLayout
       +- Pages (Login, Dashboard, BoardDetail)
          +- Features/Boards/Components
             +- BoardCard
             +- List
                +- TaskCard
                +- AddCardForm
             +- AddListForm
             +- CardModal
\end{lstlisting}

\chapter{Triển khai và Demo sản phẩm}

\section{Phân tích Luồng dữ liệu (State Flow) với Zustand}
Để hiểu rõ sức mạnh của State Management trong React, hãy phân tích chi tiết vòng đời (vòng lặp dữ liệu) khi người dùng thực hiện một thao tác đơn giản: "Thêm một Card mới".
\begin{enumerate}
    \item \textbf{Tương tác người dùng (User Action):} Người dùng mở form ở cuối cột, nhập "Thiết kế logo" vào \texttt{<textarea>} bên trong component \texttt{AddCardForm} và nhấn nút "Thêm".
    \item \textbf{Gọi Action (Dispatcher):} Thay vì xử lý logic phức tạp tại chỗ, component lấy hàm \texttt{addCard} từ \texttt{useBoardStore} và truyền các tham số cần thiết: \texttt{addCard(boardId, listId, "Thiết kế logo")}.
    \item \textbf{Xử lý nghiệp vụ (Mutation với Immer):} Bên trong \texttt{Zustand Store}, hệ thống tìm đúng \texttt{Board} theo ID, rồi tìm tiếp đúng \texttt{List} theo ID. Dưới sự hỗ trợ của thư viện \texttt{Immer}, hàm \texttt{push} được gọi một cách tự nhiên để đẩy object Card mới vào mảng \texttt{list.cards}. Ngầm bên dưới, Immer tự động tạo ra một bản sao mới hoàn toàn (Immutable Object) cho toàn bộ cây State.
    \item \textbf{Đồng bộ lưu trữ (Persist Middleware):} Ngay khi phát hiện State thay đổi, middleware \texttt{persist} tự động chuyển toàn bộ cây dữ liệu JSON và lưu đè xuống \texttt{localStorage} của trình duyệt, đảm bảo dữ liệu vĩnh viễn an toàn kể cả khi người dùng đóng máy tính.
    \item \textbf{Đối chiếu và Hiển thị (Reconciliation \& Re-render):} Zustand thông báo cho React biết State đã thay đổi. React đối chiếu (diffing) Virtual DOM cũ và Virtual DOM mới. Nó phát hiện ra chỉ có duy nhất Component \texttt{List} (nơi chứa Card mới) là có thay đổi. React ra lệnh cập nhật đúng phần tử DOM đó lên màn hình. Giao diện xuất hiện một thẻ công việc mới cực kỳ mượt mà.
\end{enumerate}

\section{Chiến lược triển khai Kéo Thả (Drag and Drop)}
Tính năng kéo thả là linh hồn của dự án nhưng cũng là bài toán kỹ thuật hóc búa nhất. Nhóm quyết định áp dụng thư viện \textbf{@dnd-kit/core} thay vì \textit{react-beautiful-dnd} cũ kỹ vì dnd-kit hỗ trợ kiến trúc mô-đun hóa, nhẹ và tương thích hoàn hảo với React Hooks.

Xử lý luồng kéo thả bao gồm 3 trạng thái sự kiện:
\begin{itemize}
    \item \textbf{onDragStart:} Nhận diện ID của thẻ đang được cầm lên (Active Card), tạo ra một bản sao mờ (overlay) di chuyển theo con trỏ chuột.
    \item \textbf{onDragOver:} Phát hiện Thẻ đang di chuyển bay ngang qua Cột (List) nào. Tính toán logic chuyển Thẻ từ Cột cũ sang Cột mới tạm thời để người dùng xem trước (Preview) vị trí rơi.
    \item \textbf{onDragEnd:} Khi người dùng nhả chuột, tính toán chính xác Vị trí chỉ số (Index) mới, gọi action \texttt{moveCard(activeId, overId, newIndex)} để chốt vị trí cập nhật vào Zustand Store.
\end{itemize}

\section{Hướng dẫn Triển khai (Deployment Guide)}
Để dễ dàng chia sẻ mã nguồn và để giảng viên nghiệm thu dự án, nhóm đã thiết lập cấu hình đóng gói (Build) cực kỳ tinh gọn:
\begin{enumerate}
    \item Yêu cầu hệ thống: Cài đặt NodeJS (v20+).
    \item \textbf{Khởi chạy cục bộ (Development):} Mở terminal tại thư mục gốc, gõ \texttt{npm install} để kéo các thư viện từ server npmJS về máy. Sau đó gõ \texttt{npm run dev} để kích hoạt máy chủ phát triển tại địa chỉ \texttt{localhost:5173}.
    \item \textbf{Đóng gói sản phẩm (Production Build):} Gõ \texttt{npm run build}. TypeScript sẽ rà soát lỗi tĩnh (\texttt{tsc -b}), sau đó Vite sẽ nén (minify) toàn bộ HTML, CSS, JS thành các file siêu nhỏ đặt trong thư mục \texttt{dist/}. Các file này đã sẵn sàng để được tải lên bất kỳ máy chủ Hosting tĩnh nào (như Vercel, Netlify, hay Github Pages) để đưa sản phẩm ra thế giới.
\end{enumerate}
